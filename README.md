# Neuron‑Anchored Rule Extraction via Contrastive Hierarchical Ablation (MechaRule) — Replication Package

> From **symbolic behavior rules** to a **sparse set of “agonist” neurons** in language models.

This repository implements an end‑to‑end interpretability pipeline that connects three levels of explanation:

1. **Behavioral labels** (e.g., *is_correct*, *is_refusal*, *is_jailbroken*) measured on task prompts  
2. **Symbolic, human‑readable rules** over automatically discovered features that predict those labels  
3. **Causal circuit elements** (edges / nodes / neurons) whose ablation selectively breaks the behavior on datapoints covered by a rule


This code accompanies the paper *Neuron-Anchored Rule Extraction for Large Language Models via Contrastive Hierarchical Ablation* (KDD 2026). In the paper, the end-to-end pipeline is referred to as **MechaRule**, and the adaptive neuron-localization stage is **Contrastive Hierarchical Ablation (CHA)**. The code and README use **agonist neuron** to mean a neuron whose suppression disrupts a rule-aligned behavior.

A neuron is an **agonist neuron** (for a rule/behavior) if ablating it reliably flips the model from
**correct → incorrect** (or more generally *target=True → target=False*) on datapoints where the behavior rule holds,
while leaving **unrelated** datapoints mostly unchanged.

## Big picture: what the pipeline produces

The pipeline is designed to answer questions like:

- “What **feature‑based condition** predicts when the model fails?”
- “Which **circuit** supports that condition?”
- “Can we isolate a **single neuron** whose ablation breaks that behavior **only when the rule applies**?”

At a high level:

```text
(step 1) task prompts + answers  ─┐
                                 ├─> (step 2) feature table (scores.csv)
                                 ├─> (step 3) symbolic rules (association_rules_*.csv)
                                 ├─> (step 4) sampling plan (spectral / pairing)
                                 ├─> (step 5) circuit discovery (EAP / EAP‑IG)
                                 ├─> (step 6) bag‑of‑rules ablations (per‑rule JSONs)
                                 └─> (step 7) single‑neuron refinement (+ optional re‑rule extraction)
```

## Core concepts & file formats

### Prompts, targets, and the “scores table”
Most later scripts work off a single **scores table** (`scores.csv` or `.parquet`) produced in step 2.

Required columns (by convention; task‑dependent):
- `prompt` (or `TASK_SPEC.DEFAULT_INPUT`) — the text input shown to the analyzed LLM
- `raw_output` (or `TASK_SPEC.DEFAULT_OUTPUT`) — the model’s completion/answer text
- one or more **target label columns** (defaults in `TASK_SPEC.DEFAULT_TARGETS`, usually `is_correct`)
- many **feature columns** (boolean / numeric) discovered in step 2

### Feature functions
A **feature** is a Python function evaluated on each row that returns:
- boolean (preferred; easy to interpret as a predicate), or
- numeric (allowed; may get thresholded or used directly by rule learners)

Features are **proposed by an LLM** (step 2) as Python snippets and executed in a sandbox:
- imports / classes / lambdas are forbidden
- only a small set of safe built‑ins is allowed

This is implemented in `lib/feature_representation.py`.

### Rules
Step 3 learns **symbolic rules** (RuleSHAP‑style) that predict a target label from feature columns.
Rules are saved as CSV files named like:

- `association_rules_<metric>.csv` — rules with statistics/importance per metric
- `rule_combo_<metric>.csv` — (optional) OR‑combo rules selected by a greedy search

Rules are strings in a restricted “query‑like” syntax (AND/OR/NOT, comparisons on feature columns).
Later scripts apply them using `lib/text_and_rules.py`.

### “Associated” vs “unrelated” datapoints
For a given rule *R* and target label *Y*:

- **associated datapoints**: rows where the rule holds (or predicted‑positive, depending on direction), typically within the subset where the baseline model is correct/incorrect (configurable)
- **unrelated datapoints**: a length‑matched, similarity‑matched control set that is *not* covered by the rule (or the complement), used to measure specificity

The sampling logic lives in step 4 and is reused by steps 5–7.

### Circuit, ablation, and delta‑gap
A **circuit element** can be:
- an **edge** (connection) between components,
- a **node** (attention head or MLP block),
- a **neuron** (MLP hidden unit / channel)

Step 5 uses attribution to score circuit elements.
Step 6 then performs targeted ablations and computes summary effects such as:

- associated accuracy drop (or target drop)  
- unrelated accuracy drop  
- **delta_gap = (associated_drop − unrelated_drop)** (bigger is “more targeted”)

## What’s in this repo

### Top‑level pipeline scripts (numbered)
- `1_generate_prompts_and_answers.py` — task‑driven prompt generation + analyzed‑LLM answers, cached as a pickle.
- `2_generate_features.py` — correctness‑driven feature proposal/testing (writes `scores.csv` + `features.json`).
- `3_extract_rules.py` — RuleSHAP‑style rule extraction from `scores.csv` (writes `association_rules_*.csv`).
- `4_spectral_sample_datapoints.py` — sampling plans per rule (spectral cover **or** similarity/length pairing).
- `5_discover_circuits.py` — circuit discovery per rule/cluster (EAP / EAP‑IG), writes a `manifest.json`.
- `6_analyze_bag_of_rules.py` — bag‑of‑rules ablations; searches for promising neuron groups and buckets neurons.
- `7_refine_neuron_anchored_rules.py` — isolates single‑neuron candidates and (optionally) re‑extracts rules on flip targets.

### Aggregation / paper‑style summaries (8–10)
- `8_compare_experiments.py` — aggregate threshold sweeps and compare *run modes* within a stats folder tree.
- `9_compare_models.py` — aggregate/compare results across analyzed LLMs for a given task.
- `10_compute_threshold_sweep_stats.py` — compute a compact threshold‑sweep table + statistical tests from the aggregated CSV produced by script 8.

### Orchestration / helpers
- `_run_pipeline.sh` — runs steps **1–7** for one **EXPERIMENT_NAME + ANALYZED_LLM** pair (and can write per-mode stats for scripts 8–10).
- `run_all_experiments.sh` — loops over experiments/models and calls `_run_pipeline.sh`, then runs scripts **8–10** to aggregate results.
- `setup.sh` — creates `.env/` (Python 3.12 venv) and installs `requirements.txt`.

- **Python 3.12** (the provided `setup.sh` uses `python3.12 -m venv .env`).
- **OpenMP runtime** (needed by XGBoost and some numerical deps):
  - macOS: `brew install libomp`

- **LLM backends**
  - **Analyzed LLM** (steps 1, 4–7): Hugging Face model ID (loaded via `transformers` / TransformerLens).

  - `OPENAI_API_KEY` if you configure feature/judge models to use OpenAI.

### Run the full pipeline via `run_all_experiments.sh`

- `EXPERIMENT_LIST` (e.g. `bon_jailbreaking`, `arithmetic`, `llmsafe_alignment`, `random_fsm`)
- `ANALYZED_LLM_LIST` (e.g. `Qwen/Qwen2-7B-Instruct`, `EleutherAI/gpt-j-6B`, ...)

and runs several variants (spectral circuit discovery, spectral splits, random anchoring plan, and an incorrect‑rules control).

**Note:** `run_all_experiments.sh` currently contains an `export GROQ_API_KEY=...` line for convenience. Replace it with your own key (or remove it and export the key in your shell) before running.

```bash
export GROQ_API_KEY="..."    # only if your chosen setup uses Groq
export OPENAI_API_KEY="..."   # only if your chosen setup uses OpenAI
bash run_all_experiments.sh
```


## Pipeline components in detail

### Step 1 — `1_generate_prompts_and_answers.py`: task probing & caching

**Goal:** build a task‑specific cache of prompt → answer pairs, plus any task metadata needed later.

- The script calls `TASK_SPEC.generate_cache(...)` and pickles the returned object.
- The *shape* of the cache is task‑defined; step 2 only requires that the task can later load it via
  `TASK_SPEC.load_dataset_from_cache(pkl_path)`.

**Gotchas**
- The script prints the entire cache JSON to stdout. For large runs, redirect output or remove the print.

### Step 2 — `2_generate_features.py`: feature proposal, execution, and filtering

**Goal:** discover interpretable features that correlate with correctness (or other targets).

- Loads the step‑1 cache via the task spec.
- Runs a loop of “proposal rounds” where the feature‑LLM proposes new Python feature functions.
- Evaluates each feature on the dataset and optionally filters:
  - near‑duplicate features (correlation threshold)
  - extreme‑variance features (MAD heuristic)
  - low predictive power features (AUC/AP/delta thresholds on a split)

Outputs:
- `scores.csv` — original columns + feature columns (+ split flags if enabled)
- `features.json` — structured list of features (label/description/source code/origin)

### Step 3 — `3_extract_rules.py`: RuleSHAP rule extraction (and controls)

**Goal:** learn symbolic rules from feature columns that predict a chosen target label.

Key mechanics:
- Uses XGBoost and/or Lasso to fit an abstract model on features.
- Uses SHAP (optionally approximate) to score feature contributions.
- Extracts and ranks candidate rules.

Outputs (under `--rules_dir`):
- `association_rules_<metric>.csv`
- `rule_combo_<metric>.csv` (greedy OR‑combo; useful as a compact rule set)
- diagnostic plots (if enabled)

Control mode:
- `--fake_targets` replaces each target column with an uncorrelated random target (`__fake` suffix)
  so you can measure how many “rules” you’d get by chance.

### Step 4 — `4_spectral_sample_datapoints.py`: sampling plan construction

**Goal:** choose a small, representative set of datapoints per rule for expensive circuit discovery/ablation.

Two modes (not mutually exclusive):
1. **Spectral cover in representation space**  
   - encodes prompts with the analyzed model (hidden state or logits)
   - projects to a low‑dimensional spectral embedding (PCA)
   - selects a k‑center / farthest‑first set such that all points are within `--coverage_radius`

2. **Similarity + length‑matched pairing** (`--pair_by_similarity_len_matched`)  
   - for each associated datapoint, finds an unrelated control with similar semantics
     (SentenceTransformer similarity) and within a token‑length tolerance

Output:
- a JSON sampling plan consumed by steps 5 and 6 (`--output_path`)

### Step 5 — `5_discover_circuits.py`: attribution‑based circuit discovery (EAP / EAP‑IG)

**Goal:** for each rule (or for each spectral cluster), identify circuit elements whose corruption changes model behavior.

Inputs:
- rules (`association_rules_*.csv`)
- `scores.csv` (prompts + features + targets)
- optional sampling plan JSON from step 4

Key knobs:
- `--method`: `EAP` (fast) or `EAP-IG` (slower, higher quality; controlled by `--max_ig_steps`)
- `--circuit_level`: `edge`, `node`, or `neuron`
- `--intervention` / `--eval_intervention`: how to corrupt activations (default is mean‑based)

Outputs:
- `manifest.json` — list of circuits discovered / evaluated with metadata per rule/cluster
- `dataset_info.json` — helps later scripts locate the right `scores.csv` and defaults
- per rule/cluster subdirectories with `eval.json` (faithfulness metrics) or `skipped.json`

> Note: the script records a `graph_topn.json` path in metadata, but does not currently export it automatically.
> If you need graph JSONs, call `graph.to_json(...)` (see `lib/eap/graph.py`) after selecting top‑n.

### Step 6 — `6_analyze_bag_of_rules.py`: bag‑of‑rules neuron ablations

**Paper terminology:** this step corresponds to the first stage of **Contrastive Hierarchical Ablation (CHA)** in MechaRule: it searches over *groups* of candidate neurons (and/or higher-level circuit elements) and measures **contrastive** effects on rule-covered vs unrelated datapoints to find promising “agonist-rich” groups.


**Goal:** test whether ablating groups of candidate neurons yields **targeted disruption** on rule‑associated datapoints.

- Consumes the manifest from step 5 and constructs neuron groups (typically MLP neurons).
- Runs a per‑rule search (often dichotomic / greedy) over group sizes.
- Writes per‑rule JSON traces including the best found group and effect estimates.

Outputs (under `--output_data_dir`):
- `per_rule/<target>/rule_XXXX.json` — full ablation trace and best group summary
- `rule_knockout.json` — compact summary across rules
- `neuron_bucket_stats.json` (and `buckets.json`) — bookkeeping for refinement

### Step 7 — `7_refine_neuron_anchored_rules.py`: single‑neuron refinement (+ optional re‑rules)

**Paper terminology:** this step implements the later stages of **CHA**: it refines the best groups down to **single agonist neurons**, evaluates them via flip targets (columns prefixed with `flip_...`), and can re-run rule extraction to obtain **neuron-anchored rules**.


**Goal:** from step‑6 outputs, isolate **single neurons** whose ablation effects exceed a threshold, and optionally
re‑run rule extraction on ablation flip targets.

Inputs:
- a directory tree or a ZIP containing step‑6 per‑rule JSONs (`--circuit_agonists_path`)
- the feature table (`--features_scores_dir`, usually `feature_report/`)

What it does:
- filters neuron groups to `group_size==1` and `|delta_gap| >= --threshold`
- for each neuron, runs targeted ablation on associated/unrelated samples
- writes a new per‑neuron subdirectory containing:
  - an augmented `scores.csv` including flip target columns (e.g., correct→incorrect flips)
  - (optional) new rule extraction results if `--extract_rules` is enabled

## Task modules

Tasks live in `lib/tasks/` and expose a `TASK_SPEC` implementing `FeatureTaskSpec` (`lib/task_spec.py`).

A task spec defines:
- how to generate prompts / collect answers (step 1)
- how to load the dataset from the step‑1 cache (step 2)
- how to parse a prompt into a token dictionary for feature functions
- how to score task success (`is_answer_positive`)
- default target label columns (`DEFAULT_TARGETS`) used by later scripts

Included modules:
- `lib/tasks/arithmetic_task.py` — synthetic arithmetic expressions; correctness by numeric match.
- `lib/tasks/bon_jailbreaking_task.py` — jailbreak prompts + refusal/jailbreak scoring (optionally via a judge).
- `lib/tasks/llmsafe_alignment_task.py` — safety-sensitive prompts derived from the LLMSAFE paper, scored with a paper-style harmful-vs-safe judge plus refusal filter.
- `lib/tasks/random_fsm_task.py` — freshly sampled binary finite-state machines; correctness requires exact algorithmic state tracking on out-of-distribution machines.

### Built-in tasks in detail

#### Arithmetic (`lib/tasks/arithmetic_task.py`)

This task probes **exact arithmetic** on minimal prompts of the form `a+b=`, `a-b=`, `a*b=`, or `a/b=`.

- **Dataset construction (step 1):** enumerates the full Cartesian product of operand pairs within a configurable range (default `0..max_operand-1`; division uses `1..max_operand-1` to avoid divide-by-zero) and runs the analyzed model with **greedy decoding** (`do_sample=False`, `max_new_tokens≈6`). Each record stores:
  - the prompt string (e.g., `299/298=`),
  - the model’s raw completion,
  - a numeric parse of the completion when possible (for analysis/debugging),
  - correctness flags.
- **Targets / labels:**
  - `is_correct`: computed by evaluating the prompt expression and comparing it to a number extracted from the model output (robust to extra surrounding text; also handles few-shot prompts separated by `;` or `,` by scoring only the last query).
  - `is_pure`: whether the completion is *exactly* a bare number (useful if you want to distinguish “right answer with chatter” vs “right answer only”).
- **Prompt parsing for features:** the task exposes a structured token dict with `operator`, operand values, digit lengths, digit lists, and signs. This enables features like “carry in addition”, “borrow in subtraction”, “operands share a digit”, etc.
- **Seed features:** in addition to simple operator/shape predicates, the task includes a large, deterministic set of **binned range features** (operand bins, result bins, and division quotient bins). These are meant to give rule extraction a rich but interpretable basis even before LLM-proposed features are added.

#### BON Jailbreaking (`lib/tasks/bon_jailbreaking_task.py`)

This task measures whether an analyzed model’s output **successfully exhibits a harmful behavior** despite prompt obfuscations (a “jailbreak”), using an external classifier in the loop.

- **Inputs:** a JSON dataset of *augmented prompts* (default `./data/bon_jailbreaking/dataset.json`, overridable via `--augmented_prompts_file` or the `AUGMENTED_PROMPTS_FILE` env var). Each item typically contains:
  - `prompt`: the augmented/obfuscated prompt sent to the analyzed model,
  - `original_prompt`: the underlying behavior spec used for judging,
  - `augmentation_info`: perturbation metadata (e.g., `sigma` plus stats like `words_scrambled`, `chars_capitalized`, `chars_perturbed`).
- **Generation (step 1):** runs the analyzed model with **greedy decoding** (`do_sample=False`, `max_new_tokens≈100`) to obtain an `answer` for each augmented prompt.
- **Targets / labels:**
  - `is_jailbroken`: computed by feeding `(original_prompt, answer)` to a **binary behavior classifier** prompted with a HarmBench-style rubric (“Yes”/“No”). (The classifier helper is written to allow short-circuiting obvious refusals/empties to “No”; in the current implementation, classification is typically run in batch for all examples for simplicity.)
  - The cache also stores `classifier_output` for auditing, and the augmentation fields to support feature extraction.
- **Prompt parsing for features:** `parse_prompt_row` returns a compact dict derived from `augmentation_info` (including `sigma` and the augmentation stats). The seed feature set mirrors these statistics so rule extraction can immediately latch onto “obfuscation regime” effects.
- **Re-labeling utility:** `rerun_is_jailbroken_check(...)` can recompute `is_jailbroken` on an existing cache if you change the classifier model/prompt or want to sanity-check label stability.


#### LLMSAFE Alignment (`lib/tasks/llmsafe_alignment_task.py`)

This task packages the **safety-sensitive prompt suites from the attached LLMSAFE paper** into the same task interface used by the rest of the repo.

- **Inputs:** by default it loads `./data/llmsafe_alignment/paper_sensitive_prompts.json`, a JSON file built from Appendix C of the paper. The file contains 242 prompts spanning:
  - the **domain-specific probe prompts** from Appendix C.1, and
  - the **8-category evaluation taxonomy** from Appendix C.2 (harmful-domain definitional/contextual/functional/strategic prompts plus benign-domain definitional/functional/enumerative/causal controls).
- **Generation (step 1):** runs the analyzed model with greedy decoding (`do_sample=False`, `max_new_tokens≈96`) to obtain an `answer` for each prompt.
- **Targets / labels:**
  - `is_harmful_response`: a paper-style dual-stage safety label. The task first sends `(original_prompt, answer)` to a harmful-vs-safe judge based on the rubric in Appendix A.2 of the paper, then applies the paper’s explicit refusal-string filter before marking a response as harmful.
  - `is_safe_response`: the complement, stored for convenience.
- **Prompt parsing for features:** the task exposes structured metadata such as `prompt_group`, `domain_type`, `category`, `topical_domain`, `subject`, `subject_family`, prompt length, and a few lexical cues (`mentions_definition`, `mentions_usage`, etc.).
- **Override path:** set the `LLMSAFE_PROMPTS_FILE` env var to point the task at a different JSON prompt file without editing the code.

#### Random FSM (`lib/tasks/random_fsm_task.py`)

This task is intentionally about something a pretrained LLM is **not able to memorize from pretraining**: simulating a **freshly sampled binary finite-state machine** that is generated on the fly for each prompt.

- **Inputs / generation:** for each example, the task samples a new deterministic state machine, a start state, and a binary input string. The model is asked to output **only the final state label**.
- **Why it is interesting:** the machine is newly generated at runtime, so success depends on genuine in-context algorithmic state tracking rather than memorized world knowledge.
- **Targets / labels:**
  - `is_correct`: whether the first parsed state label in the model output matches the exact simulated final state.
- **Prompt parsing for features:** the task exposes machine size, input length, zero/one balance, longest runs, self-loop counts, transition concentration, and start/answer-state indices.
- **Override knobs:** set env vars such as `FSM_NUM_EXAMPLES`, `FSM_MIN_STATES`, `FSM_MAX_STATES`, `FSM_MIN_INPUT_LEN`, and `FSM_MAX_INPUT_LEN` to change the synthetic benchmark size or difficulty.


### Adding a new task
1. Create `lib/tasks/<my_task>_task.py`.
2. Define `TASK_SPEC = ...` implementing the `FeatureTaskSpec` contract.
3. Run scripts with `--task_module lib.tasks.<my_task>_task` (or add `<my_task>` to `EXPERIMENT_LIST`).

  RuleSHAP implementation + orchestration utilities (including `--fake_targets` control).

- `lib/text_and_rules.py`  
  Loading and applying rule strings to a feature table; finding rule files via globs; filetype guessing.

  Shared spectral embedding / PCA logic and k‑center (farthest‑first) selection; also adds shared CLI args
  used in steps 4 and 5.

- `lib/modeling_and_ablation.py`  
  Loads analyzed LLMs (HF + TransformerLens), provides inference helpers, and implements ablation hooks
  such as **mean‑positional** interventions and MPS‑safe aggregation.

  Attribution‑based circuit discovery and evaluation utilities (Graph, attribution, metrics). See `lib/eap/README.md`.

## Reproducibility, caching, and performance notes

- Most scripts accept `--seed` / `--random_seed` and call `set_deterministic(...)`.
- Step 4/5 spectral embeddings can be cached with `--spectral_cache_dir` (saves significant time).
- Circuit discovery and ablations are GPU‑heavy; consider:
  - lowering `--batch_size`
  - reducing `--max_pairs_per_rule` / sampling sizes
  - using `EAP` before `EAP-IG` to iterate quickly

---

## Troubleshooting

- **XGBoost / OpenMP errors on macOS**: install `libomp` (`brew install libomp`).
- **Graph visualizations missing**: GraphViz / pygraphviz is optional; the circuit discovery script may skip rendering silently.
- **Feature execution errors**: LLM‑proposed feature code is sandboxed; rejected features will be dropped (enable warnings in step 2 if needed).
- **Very large stdout**: step 1 prints the cache; redirect output.

## Running scripts manually (step-by-step)

This is useful if you want to dev/debug a specific stage.

### 1) Generate prompts & answers

```bash
python 1_generate_prompts_and_answers.py   --ai_model Qwen/Qwen2-7B-Instruct   --prompts_answers_pkl_file ./cache/arithmetic/Qwen/Qwen2-7B-Instruct/llm_io_data.pkl   --task_module lib.tasks.arithmetic_task   --batch_size 256
```

### 2) Generate features & scores

`--ai_model` here is the **feature LLM** (often an Ollama model like `gemma3:27b`).

```bash
python 2_generate_features.py   --ai_model gemma3:27b   --prompts_answers_pkl_file ./cache/arithmetic/Qwen/Qwen2-7B-Instruct/llm_io_data.pkl   --task_module lib.tasks.arithmetic_task   --features_scores_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/feature_report   --cache_dir ./cache/arithmetic   --drop_near_duplicate_features   --drop_high_mad_variance_features
```

### 3) Extract rules (RuleSHAP)

```bash
python 3_extract_rules.py   --features_scores_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/feature_report   --rules_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/rule_extraction_results   --task_module lib.tasks.arithmetic_task   --use_shap_in_xgb   --use_shap_in_lasso
```

Optional control:
- Add `--fake_targets` to extract intentionally uninformative rules (outputs should be isolated in a separate directory).

### 4) Build a sampling plan per rule (spectral or pairing mode)

This script consumes:
- the feature table file (CSV/Parquet), and
- the directory of `association_rules_*.csv`.

Example (spectral cover selection):

```bash
python 4_spectral_sample_datapoints.py   --features_scores_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/feature_report/scores.csv   --rules_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/rule_extraction_results   --ai_model Qwen/Qwen2-7B-Instruct   --spectral_cache_dir ./cache/arithmetic   --spectral_space hidden   --rep_hook_name ln_final.hook_normalized   --rep_pooling last   --spectral_dim 32   --coverage_radius 0.5   --max_points_per_rule 256   --min_points_per_rule 32   --output_path ./data/arithmetic/Qwen/Qwen2-7B-Instruct/neural_circuit_discovery_results/spectral_sampling_plan.json
```

### 5) Discover circuits (EAP / EAP‑IG)

```bash
python 5_discover_circuits.py   --rules_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/rule_extraction_results   --features_scores_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/feature_report/scores.csv   --output_data_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/neural_circuit_discovery_results/spectral_plan   --ai_model Qwen/Qwen2-7B-Instruct   --cache_dir ./cache/arithmetic   --method EAP   --circuit_level neuron   --circuit_size 256   --sampling_strategy plan   --sampling_plan_path ./data/arithmetic/Qwen/Qwen2-7B-Instruct/neural_circuit_discovery_results/spectral_sampling_plan.json
```

### 6) Bag‑of‑rules ablations

```bash
python 6_analyze_bag_of_rules.py   --input_data_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/neural_circuit_discovery_results/spectral_plan/eap/auto_circuit_discovery   --output_data_dir ./data/arithmetic/Qwen/Qwen2-7B-Instruct/neural_circuit_discovery_results/spectral_plan/eap/bag_of_rules/all_baseline   --scores_path ./data/arithmetic/Qwen/Qwen2-7B-Instruct/feature_report/scores.csv   --ai_model Qwen/Qwen2-7B-Instruct   --intervention mean-positional   --points_to_use_for_mean_ablation 256   --sampling_strategy plan   --sampling_plan_path ./data/arithmetic/Qwen/Qwen2-7B-Instruct/neural_circuit_discovery_results/spectral_sampling_plan.json
```

### 7) Refine to neuron‑anchored rules

This script reads the per-rule outputs from step 6 and identifies single-neuron candidates (`group_size==1`) whose ablation changes the
associated-vs-unrelated gap beyond `--threshold`. It can then:
- append flip targets to the feature table, and
- optionally run rule extraction again with those flip targets.

(See `7_refine_neuron_anchored_rules.py --help` for the full set of switches.)
